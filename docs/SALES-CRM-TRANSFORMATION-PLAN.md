# WeSales AI客户雷达与轻量 CRM 产品规划

## 1. 产品定位

WeSales 是**基于微信聊天记录的客户识别、销售机会发现与轻量 CRM 记录工具**。

本次规划不是把 WeSales 做成重型企业 CRM，也不是复制竞品整套产品，而是参考竞品“AI客户雷达”在信息架构、页面模块、指标体系、视觉方向和业务组织方式上的优点，形成 **WeSales 自己的产品实现方案**。

WeSales 的核心价值应明确为：

- 以微信聊天记录为主数据源，帮助销售识别客户、整理上下文、发现机会。
- 将“聊天分析”升级为“客户识别 + 销售机会发现 + 跟进记录沉淀”。
- 提供轻量 CRM 记录能力，但保持本地优先、轻量可落地。
- 以“建议、记录、提醒、导出”为主，不承诺微信侧自动执行动作。

一句话定位：

> WeSales 是一个围绕微信聊天记录工作的 AI 客户雷达与轻量 CRM，帮助销售识别重点客户、发现销售机会、记录跟进过程，并沉淀可复盘的客户资产。

## 2. 规划前提与竞品借鉴原则

### 2.1 规划前提

- 产品名称统一为 **WeSales**，不使用 WeFlow。
- 当前能力基础是：同步 / 读取聊天记录、联系人与群聊数据，并做分析和展示。
- 当前不具备微信侧写操作能力，因此所有产品设计必须服从单向同步边界。
- 现阶段优先做产品规划与技术分工，不在本轮直接实现源码。

### 2.2 竞品借鉴原则

竞品截图体现出一套较完整的“AI客户雷达”工作流，覆盖工作台、好友分析、客户详情 / 客户360、群分析中心、销售机会中心、群详情、AI助手、客户关系地图、个人中心等页面。WeSales 可以借鉴其：

- 以工作台承接高优先级任务和 AI 洞察。
- 以客户、群、机会、关系网络组成主信息架构。
- 以指标卡、列表、详情页、时间线、画像面板组织业务。
- 以“建议下一步动作”的方式帮助销售推进工作。

但 WeSales 不应：

- 使用竞品品牌名、专有文案、视觉细节或宣称完全复制。
- 设计任何超出 WeSales 当前数据边界的自动化能力。
- 将“分析建议”包装成“系统自动代替销售执行”。

## 3. 单向同步能力边界

本章节是产品规划与实现的约束前提，所有页面、文案、交互和验收都必须遵守。

### 3.1 允许做什么

- 同步、读取、解析微信聊天记录。
- 同步、读取联系人、群聊、群成员、聊天时间、消息内容、附件元数据等只读信息。
- 基于聊天记录做客户识别、意向判断、沉默检测、关系分析、机会识别。
- 在 WeSales 本地生成客户档案、跟进记录、任务、标签、备注、AI 洞察和分析快照。
- 提供复制话术、复制手机号、打开原始会话、查看证据消息、导出材料等辅助动作。
- 基于规则或 AI 生成建议计划、任务草稿、演示计划、案例推荐和待办提醒。

### 3.2 不允许做什么

- 不能发送消息给客户。
- 不能自动回复客户。
- 不能添加好友、删除好友或修改好友关系。
- 不能修改微信侧备注、标签、群信息或聊天数据。
- 不能执行微信侧动作，例如发消息、拨号、群发、转发、分享资料。
- 不能承诺“自动触达”“自动跟进”“自动发送给客户”“一键联系客户”等能力。

### 3.3 边界下的动作降级策略

竞品中涉及执行动作的模块，需要在 WeSales 中统一降级为“建议 + 记录 + 跳转 + 复制”：

- 发消息 -> 复制建议话术 / 打开原始会话 / 创建跟进任务
- 打电话 -> 复制手机号 / 记录电话跟进 / 创建电话任务
- 立即联系 -> 查看客户 / 创建跟进 / 标记待处理
- 发送案例 -> 生成案例推荐 / 复制话术 / 保存素材
- 安排演示 -> 创建演示任务 / 生成演示计划
- 一键生成跟进计划 -> 只生成计划和任务草稿，不执行触达

## 4. 目标用户与核心场景

### 4.1 目标用户

- 个人销售：微信是主要沟通阵地，需要高频查看客户状态和跟进节奏。
- 销售主管：需要了解客户覆盖、机会分布、团队跟进风险和重点客户。
- 小型销售团队 / 私域销售团队：缺少贴近微信场景的轻量 CRM。
- 顾问式销售 / 客户成功：需要长期跟进、多轮沟通和上下文沉淀。

### 4.2 核心场景

- 从聊天记录中识别潜在客户、高意向客户、沉默客户和风险客户。
- 将联系人、群聊、消息证据、跟进记录和机会信息串成客户视图。
- 基于聊天上下文生成跟进建议、任务草稿和销售机会线索。
- 在本地记录客户状态、跟进动作、关键节点和复盘依据。
- 通过工作台集中查看“今天先处理谁、为什么处理、建议怎么处理”。

## 5. 信息架构重组

参考竞品的页面组织方式，WeSales 建议采用以下一级模块。

### 5.1 一级模块

1. 工作台
2. 好友分析
3. 客户360
4. 群分析
5. 销售机会
6. 销售时间轴
7. 关键人群
8. AI助手
9. 客户地图
10. 数据导出
11. 个人中心

### 5.2 模块定位说明

| 一级模块 | WeSales 定位 | 主要输出 |
|---|---|---|
| 工作台 | 销售当日决策入口 | 今日待处理客户、重点机会、风险提醒、AI建议 |
| 好友分析 | 单个联系人与客户识别入口 | 好友画像、意向等级、互动热度、客户转化建议 |
| 客户360 | 客户经营主视图 | 客户档案、联系人、消息证据、任务、跟进、机会 |
| 群分析 | 群聊识别与经营分析 | 群活跃度、关键成员、群主题、机会线索 |
| 销售机会 | 机会识别与推进记录 | 机会列表、阶段、优先级、阻塞点、下一步计划 |
| 销售时间轴 | 经营过程回放 | 消息证据、跟进记录、任务完成、机会变化 |
| 关键人群 | 按业务规则聚合重点对象 | 高意向、待跟进、沉默风险、已成交、可转介绍 |
| AI助手 | 建议和总结中心 | 机会解释、跟进建议、案例推荐、计划草稿 |
| 客户地图 | 地区与关系分布视图 | 地区识别、客户聚集、关系链路、区域机会 |
| 数据导出 | 数据留档和复盘出口 | 客户包、机会包、时间轴证据包、分析导出 |
| 个人中心 | 个人偏好与数据管理 | 账号、本地数据、模型配置、规则配置 |

### 5.3 页面组织原则

- 工作台承担“先做什么”的决策，不承担全部详情。
- 好友分析、客户360、群分析分别承接“人、客户、群”三类经营对象。
- 销售机会和销售时间轴分别承接“推进状态”和“过程证据”。
- AI助手负责生成建议，不直接替代用户执行动作。
- 客户地图和关键人群承担高级分析与筛选，不应成为首期硬依赖。

## 6. 核心页面规划

### 6.1 工作台

目标：打开 WeSales 后立即看到高优先级工作。

建议模块：

- 今日待处理客户
- 新识别高意向客户
- 沉默风险客户
- 待补充跟进记录
- 新发现销售机会
- 群聊机会提醒
- 本周跟进任务概览

建议动作：

- 查看客户
- 打开原始会话
- 复制建议话术
- 创建跟进
- 创建任务
- 标记待处理 / 已处理

### 6.2 好友分析

目标：把“联系人”升级为“可识别的销售对象”。

建议模块：

- 基础资料与标签
- 意向等级与理由
- 最近互动热度
- 关键词与需求信号
- 最近消息证据
- 推荐客户归类

建议输出：

- 是否建议纳入客户池
- 是否需要补记跟进
- 是否存在沉默风险
- 是否推荐转入销售机会

### 6.3 客户360

目标：成为 WeSales 的核心详情页。

建议分区：

- 客户概览
- 联系人与角色
- 会话入口
- 跟进记录
- 任务列表
- 销售机会
- 关键消息证据
- AI洞察
- 相关群聊
- 附件与素材

建议强调：

- 所有结论尽量回链到聊天消息证据。
- 不展示“立即发送”类按钮，统一替换为查看、复制、记录、创建任务。

### 6.4 群分析

目标：识别群聊中的商机线索与关键人。

建议模块：

- 群活跃度趋势
- 核心发言人
- 群主题与讨论重点
- 高潜成员识别
- 群内机会提醒
- 群详情入口

建议动作：

- 查看群详情
- 查看群内关键成员
- 将群成员加入关键人群
- 从群消息创建机会线索

### 6.5 销售机会

目标：提供轻量机会管理，不引入重 CRM 流程负担。

建议视图：

- 机会列表
- 机会阶段看板
- 机会详情抽屉

建议字段：

- 机会名称
- 关联客户
- 来源会话 / 群聊
- 当前阶段
- 意向分级
- 预计金额区间
- 阻塞点
- 下一步计划
- 最近证据消息

### 6.6 销售时间轴

目标：把聊天、跟进、任务、机会变化串成连续过程。

建议事件：

- 关键消息
- 客户状态变化
- 新建 / 更新机会
- 创建任务
- 完成任务
- 记录电话 / 见面 / 演示跟进
- AI洞察生成

价值：

- 便于复盘客户经营过程。
- 便于主管查看推进逻辑。
- 便于导出证据链。

### 6.7 关键人群

目标：为筛选、复盘和批量处理提供业务视角。

建议分组：

- 高意向待跟进
- 最近沉默客户
- 多次提及价格客户
- 近期活跃群关键成员
- 已成交可复购客户
- 可转介绍客户

### 6.8 AI助手

目标：让 AI 成为建议生成器，而不是动作执行器。

建议能力：

- 生成客户摘要
- 解释意向判断依据
- 生成跟进计划草稿
- 生成电话沟通提纲
- 生成演示计划
- 生成案例推荐
- 生成话术建议

输出形式：

- 建议文本
- 任务草稿
- 跟进建议
- 可复制话术
- 待人工确认的结构化字段

### 6.9 客户地图

目标：把地区识别和关系网络可视化，提升区域经营视角。

建议模块：

- 地区分布
- 城市聚类
- 区域机会热度
- 客户关系网络
- 群与客户关联关系

### 6.10 数据导出

目标：支持留档、复盘、汇报和外部继续处理。

建议导出对象：

- 客户档案
- 客户360摘要
- 销售机会清单
- 跟进记录
- 销售时间轴证据包
- 群分析摘要
- 关键人群列表

### 6.11 个人中心

目标：承接本地化配置和个人使用偏好。

建议模块：

- 数据源管理
- 本地备份与恢复
- 模型配置
- AI提示词与规则配置
- 导出设置
- 隐私与安全说明

## 7. 数据模型草案

保留原有“本地优先、业务表与原始聊天表分层”的思路，但按 WeSales 模块重组为以下实体。

| 实体 | 关键字段 | 说明 |
|---|---|---|
| FriendProfile | `id`, `contactUsername`, `displayName`, `avatar`, `tags`, `intentLevel`, `relationshipType`, `lastContactAt`, `silenceDays`, `updatedAt` | 好友画像，承接联系人销售识别 |
| Customer360 | `id`, `customerName`, `status`, `tier`, `source`, `primaryContactId`, `ownerId`, `lastContactAt`, `nextFollowUpAt`, `updatedAt` | 客户主体，承接客户经营视图 |
| GroupProfile | `id`, `groupId`, `groupName`, `memberCount`, `topicSummary`, `activityScore`, `keyPeopleSnapshot`, `updatedAt` | 群画像，用于群分析和群机会识别 |
| SalesOpportunity | `id`, `customerId`, `sourceType`, `sourceId`, `title`, `stage`, `priority`, `amountRange`, `winProbability`, `blockers`, `nextPlan`, `updatedAt` | 销售机会对象 |
| FollowUpRecord | `id`, `customerId`, `opportunityId`, `relatedContactUsername`, `relatedSessionId`, `type`, `summary`, `result`, `nextAction`, `createdAt` | 跟进记录，承接人工沉淀 |
| Task | `id`, `entityType`, `entityId`, `title`, `taskType`, `priority`, `dueAt`, `status`, `source`, `createdAt`, `updatedAt` | 任务表，支持客户、机会、群聊关联 |
| MessageEvidence | `id`, `messageId`, `sessionId`, `customerId`, `opportunityId`, `evidenceType`, `snippet`, `evidenceScore`, `createdAt` | 消息证据，支撑 AI 判断可追溯 |
| AIInsight | `id`, `entityType`, `entityId`, `insightType`, `title`, `summary`, `reasoning`, `confidence`, `suggestedActions`, `status`, `createdAt` | AI洞察表，强调建议性质 |
| ActivitySnapshot | `id`, `entityType`, `entityId`, `messageCount7d`, `messageCount30d`, `lastInboundAt`, `lastOutboundAt`, `silenceDays`, `responseMedianMinutes`, `updatedAt` | 活动快照，用于列表和指标 |
| RegionRecognition | `id`, `entityType`, `entityId`, `country`, `province`, `city`, `confidence`, `source`, `updatedAt` | 地区识别表，服务客户地图 |

### 7.1 建模原则

- 原始聊天数据与业务记录分层存储。
- `AIInsight`、`intentLevel`、`winProbability` 等 AI 推断字段必须允许人工修正。
- 所有判断尽量能回链 `MessageEvidence`。
- 聚合指标使用 `ActivitySnapshot` 异步生成，避免直接扫描消息表。
- 首期优先本地表落地，后续再考虑同步或共享。

## 8. 三步实施路线

### 8.1 第一步：WeSales AI客户雷达基础版

目标：先建立“客户识别 + 工作台 + 客户基础视图”的产品骨架，让 WeSales 从聊天分析工具转向 AI客户雷达。

范围：

- 工作台基础版
- 好友分析基础版
- 客户360基础版
- 关键人群基础版
- 销售时间轴基础版
- AI洞察基础生成
- 数据导出基础能力

工作分工：

- 产品/交互
  - 定义工作台卡片结构、优先级规则和详情跳转路径。
  - 明确好友分析、客户360、销售时间轴三类页面的信息层级。
  - 统一替换所有超边界文案，例如“自动跟进”“发送给客户”。
- 前端页面与组件
  - 规划工作台指标卡、列表卡、证据卡、建议动作卡等基础组件。
  - 规划好友分析页、客户360页、时间轴页的通用详情布局。
  - 规划只读证据抽屉、复制话术弹层、创建记录弹层。
- 数据模型/本地表
  - 落地 `FriendProfile`、`Customer360`、`MessageEvidence`、`AIInsight`、`ActivitySnapshot` 基础结构。
  - 建立联系人、会话、消息到客户对象的映射关系。
  - 预留 `status`、`intentLevel`、`source`、`confidence` 等人工修正字段。
- 服务层/API/IPC
  - 定义联系人画像、客户详情、消息证据、活动快照读取接口。
  - 定义创建客户、更新客户状态、写入 AI洞察状态的本地 API / IPC。
  - 统一详情页查询协议，避免每页重复拉取多份聚合数据。
- 规则/AI分析
  - 先用规则 + 轻量 AI 结合做意向等级、沉默风险、近期活跃识别。
  - 输出“判断依据 + 建议动作”，不输出可自动执行动作。
  - 为高意向、沉默风险、待补跟进三类洞察定义最小规则集。
- 验收标准
  - 用户能在工作台看到高意向、沉默风险、待处理客户。
  - 用户能从好友分析进入客户360，并看到消息证据与基础建议。
  - 所有建议动作都符合单向同步边界。
  - 页面中的结论能回链到消息证据或活动指标。

### 8.2 第二步：销售机会与跟进记录闭环

目标：补齐机会、跟进、任务三件套，让 WeSales 从“识别雷达”升级到“可记录、可推进、可复盘”的轻量 CRM。

范围：

- 销售机会中心
- 跟进记录闭环
- 任务与提醒
- 销售时间轴增强
- 群分析中心基础版

工作分工：

- 产品/交互
  - 明确机会阶段、任务状态、跟进记录模板和时间轴事件模型。
  - 将竞品中的执行型动作全部降级为建议型动作。
  - 设计“从消息证据创建机会 / 跟进 / 任务”的最短路径。
- 前端页面与组件
  - 规划机会列表、阶段看板、机会详情抽屉。
  - 规划跟进记录表单、任务表单、时间轴事件卡。
  - 规划群分析中心与群详情基础页面，支持群机会线索展示。
- 数据模型/本地表
  - 落地 `SalesOpportunity`、`FollowUpRecord`、`Task`、`GroupProfile`。
  - 建立客户、机会、任务、跟进、消息证据间的外键关系。
  - 为机会阶段、任务类型、跟进结果建立稳定枚举。
- 服务层/API/IPC
  - 定义机会 CRUD、跟进记录 CRUD、任务 CRUD、本地筛选查询接口。
  - 定义从消息证据派生机会和跟进的转换接口。
  - 定义群画像、群详情、群机会线索读取接口。
- 规则/AI分析
  - 识别价格、方案、演示、采购、决策人等机会信号。
  - 生成机会摘要、阻塞点、下一步计划草稿。
  - 为“电话跟进”“演示安排”“案例发送”输出任务草稿和文案建议，不执行触达。
- 验收标准
  - 用户可从消息证据创建销售机会、跟进记录和任务。
  - 销售时间轴能展示机会变化和人工跟进过程。
  - 群分析可识别群内关键成员和机会线索。
  - 所有“下一步动作”都以记录、复制、跳转、建任务为主。

### 8.3 第三步：AI助手、客户地图与高级分析

目标：在基础闭环稳定后，增强 AI 建议质量、客户关系可视化和分析深度。

范围：

- AI助手完整化
- 客户地图
- 地区识别
- 高级关键人群
- 高级群分析
- 高级导出与复盘

工作分工：

- 产品/交互
  - 定义 AI助手的输入输出边界，明确“建议可采纳、动作不自动执行”。
  - 设计客户地图和关系网络的筛选方式、解释方式和使用场景。
  - 设计高级分析页中的区域分布、机会漏斗、客户分层和经营风险视图。
- 前端页面与组件
  - 规划 AI 助手对话面板、建议卡、计划草稿卡、案例推荐卡。
  - 规划地图视图、关系网络视图、区域筛选器、漏斗和热力模块。
  - 规划高级导出面板，支持客户包、证据包、复盘包。
- 数据模型/本地表
  - 落地 `RegionRecognition` 并扩展 `AIInsight`、`ActivitySnapshot`。
  - 补充关系网络所需的联系人-客户-群聊关联快照。
  - 为高级导出增加模板配置和导出任务元数据。
- 服务层/API/IPC
  - 定义 AI助手问答、摘要生成、计划草稿生成接口。
  - 定义地图和关系网络所需聚合查询接口。
  - 定义高级分析和导出任务的本地计算 / IPC 协议。
- 规则/AI分析
  - 引入更稳定的多维评分：意向、活跃、风险、成交推进、复购潜力。
  - 基于聊天证据生成客户摘要、演示计划、电话提纲、案例推荐。
  - 做地区识别、关键人关系推断和群影响力分析，但统一保留置信度与证据说明。
- 验收标准
  - AI助手可生成可解释、可复制、可转任务的建议结果。
  - 客户地图可展示地区识别结果和基础关系网络。
  - 高级分析可支持客户分层、区域机会、群影响力、复购潜力等视角。
  - 所有 AI 输出均为建议或草稿，不涉及自动触达和微信侧动作。

## 9. 文案与交互约束

为保证产品表达与能力一致，以下措辞需要统一删除或替换：

- 删除：自动触达、自动跟进、自动发送给客户、自动联系、自动加好友
- 替换为：生成建议、复制话术、创建任务、记录跟进、打开原始会话、查看客户、标记待处理

页面按钮命名建议：

- 不用“立即发送”，改为“复制建议话术”
- 不用“现在联系”，改为“打开原始会话”
- 不用“一键安排电话”，改为“创建电话任务”
- 不用“安排演示并通知客户”，改为“生成演示计划”

## 10. 非目标

首轮规划暂不覆盖以下内容：

- 微信侧自动执行能力
- 企业级 CRM 权限体系
- 合同、审批、回款等重流程模块
- 大规模营销自动化
- 全渠道客户数据中台
- 第三方 CRM 双向深度同步

## 11. 风险与注意事项

### 11.1 能力边界风险

- 如果页面设计过度借鉴竞品动作层，会自然滑向“自动执行”错觉。
- 必须在每个功能点上检查是否触碰微信侧写操作边界。

### 11.2 数据质量风险

- 联系人和群成员命名可能不稳定，客户归并需要允许人工修正。
- 聊天内容中的意向判断可能误判，必须保留证据与置信度。

### 11.3 性能风险

- 工作台、关键人群、客户地图和高级分析都依赖聚合指标。
- 需要通过快照表和异步计算控制列表性能。

### 11.4 隐私与合规风险

- 聊天记录、客户状态、机会信息、跟进记录均为敏感数据。
- 本地优先、可追溯、可导出范围控制必须保持为基础原则。

## 12. 总体验收口径

- 用户能明确理解 WeSales 是 AI客户雷达与轻量 CRM，而不是单纯聊天查看器。
- 产品架构围绕工作台、好友分析、客户360、群分析、销售机会、销售时间轴、关键人群、AI助手、客户地图、数据导出、个人中心展开。
- 所有功能描述都符合单向同步边界，不出现自动触达、自动发送、自动联系类承诺。
- 机会、跟进、任务、时间轴和证据链能形成可落地的轻量 CRM 闭环。
- 竞品借鉴体现在业务组织方式与模块设计上，而不是品牌、专有文案或 UI 复制。
